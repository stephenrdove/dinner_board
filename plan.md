# DinnerBoard — Project Spec

A lightweight family meal planning web app with real-time sync via Cloudflare Workers + KV. Hosted on GitHub Pages. Used by two adults to coordinate a fortnightly meal plan and shared grocery list.

---

## Context

- **Users:** Dad (primary), Mom (occasional viewer + grocery checker)
- **Family:** 2 adults + 1 toddler (2.5 years old)
- **Dietary constraint:** Mom has celiac — everything must be **strictly gluten-free**
- **Meal cadence:** Fortnightly planning (14 days at a time), with 2 leftover nights built in per cycle
- **Meal plan source:** JSON generated by a Claude prompt, imported via the app's Import tab

---

## Architecture

```
GitHub Pages (index.html)
        │
        │  GET /state
        │  PUT /state
        ▼
Cloudflare Worker (worker.js)
        │
        ▼
Cloudflare KV (namespace: DINNERBOARD)
   key: "dinnerboard_state"
   value: JSON blob { meals, grocery, theme }
```

### Key decisions
- **Single shared state blob** — the entire app state is one KV value. Simple, no migrations needed.
- **Auth via shared secret** — `X-Auth-Token` header checked against `AUTH_TOKEN` Worker secret. Not OAuth, intentionally simple.
- **Config stored in localStorage per device** — Worker URL and auth token are device-local (intentional — don't sync credentials through KV).
- **Debounced writes** — changes are saved 800ms after the last user action to avoid excessive KV writes.
- **Fallback to localStorage** — if Worker is unreachable, app falls back to `dinnerboard_local` in localStorage silently.

## State Schema

```typescript
interface AppState {
  meals: Record<string, Meal>;   // keyed by "YYYY-MM-DD"
  grocery: GroceryItem[];
  theme: 'light' | 'dark';
}

interface Meal {
  name: string;
  notes: string;
  leftover: boolean;
}

interface GroceryItem {
  id: string;           // "g_" + timestamp + random
  name: string;
  category: GroceryCategory;
  checked: boolean;
  warn: boolean;        // true = must buy certified GF version
}

type GroceryCategory =
  | 'Produce'
  | 'Meat & Seafood'
  | 'Dairy & Eggs'
  | 'Frozen'
  | 'Pantry / Dry Goods'
  | 'Canned Goods'
  | 'Condiments & Sauces'
  | 'Other';
```

## Import JSON Schema

Every two weeks, a Claude prompt generates a meal plan as JSON. The user pastes it into the Import tab. Schema:

```json
{
  "weeks": [
    {
      "week": 1,
      "days": [
        {
          "date": "2026-04-27",
          "meal": "Al Fresco Chicken Meatballs with GF Pasta",
          "notes": "Double batch — leftovers Tuesday",
          "leftover": false
        }
      ]
    },
    {
      "week": 2,
      "days": [...]
    }
  ],
  "grocery": [
    { "category": "Produce", "name": "Broccoli", "warn": false },
    { "category": "Condiments & Sauces", "name": "Tamari (soy sauce)", "warn": true }
  ]
}
```

**Import behavior:**
- Merges meals into existing state (does not wipe meals outside the imported date range)
- Appends grocery items (does not clear existing list)
- Redirects to Calendar view after 1.2s on success

## Worker API

**Base URL:** `https://dinnerboard.<yourname>.workers.dev`

| Method | Path | Description |
|--------|------|-------------|
| GET | `/state` | Returns current app state JSON |
| PUT | `/state` | Saves new app state JSON |
| OPTIONS | `*` | CORS preflight |

**Auth:** All requests require `X-Auth-Token: <secret>` header. Returns `401` if missing or wrong.

**CORS:** `Access-Control-Allow-Origin: *` (app is served from a different origin than the worker)

**KV:** Single key `dinnerboard_state`, value is the full state JSON string.

## Frontend Views

### Calendar
- 2-week grid (7 columns × 2 rows of days)
- Navigation: Prev / Next fortnight buttons, "This Fortnight" resets to current period
- Today highlighted with accent background
- Click any day → modal to set/edit meal name, notes, leftover flag
- Leftover days shown in green italic with ↩ prefix
- Empty days show "tap to add" placeholder

### Groceries
- Items grouped by category with category headers
- Checkboxes to mark items bought (strikethrough when checked)
- ⚠️ badge on items needing certified GF versions
- Delete button per item (appears on hover)
- "Clear checked" button removes bought items
- "Clear all" with confirmation bar
- Manual add form: category select + text input + GF flag checkbox

### Import
- Textarea for raw JSON paste
- Validate and import button
- Status message (success count or error detail)
- Schema reference shown inline

### Settings (⚙️)
- Worker URL input
- Auth token input (password field)
- "Save & Test Connection" — hits GET /state to verify, then saves config to localStorage
- "Clear Config" — removes config, shows banner
- Setup checklist shown inline

## UI / Design

- **Fonts:** Fraunces (serif, display/headings) + DM Sans (body)
- **Theme:** Light/dark toggle, persisted in synced state
- **Color accent:** Warm amber `#c17f47`
- **Sync indicator:** Fixed bottom-right pill — shows spinning dot while saving, green ✓ for 2.5s after save, red with retry tap on error
- **Config banner:** Shown at top when Worker not configured, links to Settings tab
- **Loading overlay:** Full-screen on init while fetching from Worker, fades out

## Worker Setup (one-time)

```bash
# 1. Create KV namespace
wrangler kv namespace create DINNERBOARD
# → copy the id into wrangler.toml

# 2. Set auth token secret
wrangler secret put AUTH_TOKEN
# → type your secret when prompted

# 3. Deploy
wrangler deploy
# → note your Worker URL
```

**wrangler.toml:**
```toml
name = "dinnerboard"
main = "worker.js"
compatibility_date = "2024-01-01"

[[kv_namespaces]]
binding = "DINNERBOARD"
id = "REPLACE_WITH_YOUR_KV_NAMESPACE_ID"
```

## GitHub Pages Setup

1. Create repo (e.g. `dinnerboard`)
2. Add `index.html` (renamed from `meal-planner-v2.html`), `worker.js`, `wrangler.toml`
3. Settings → Pages → Deploy from branch → `main` → `/ (root)`
4. Live at: `https://<yourusername>.github.io/dinnerboard`
